# D4C Clothing Shop — Issues Log

**Date:** 2026-05-22
**Scope:** Checkout / Payment flows, RabbitMQ integration, Frontend PaymentPage

---

## CRITICAL (6 issues)

### 1. Stock deduction partial failure — no rollback

**Location:** `OrderService.java:202-208`

**Problem:** `deductStockForOrder()` calls `ProductClient.deductStock()` sequentially for each item. If any call fails (e.g., item #3 out of stock), the `@Transactional` method rolls back the OrderService DB transaction, but stock already deducted for earlier items in DynamoDB is NOT restored.

**Impact:** Stock leak — items appear reduced in inventory but no order was created.

**Fix:** Wrap `deductStockForOrder` in a try-catch that restores stock for all previously-deducted items on failure. Better: create a single batch-deduct endpoint in ProductService that handles all items atomically.

---

### 2. Webhook arrives after payment EXPIRED — user paid but order not fulfilled

**Location:** `PaymentService.java:137-150` → `markAsPaid` uses `WHERE status = 'PENDING'`

**Problem:** User scans QR and transfers money. SePay webhook arrives late (after the 5-minute expiry). `PaymentExpiryJob` has already set status to `EXPIRED`. `markAsPaid` returns 0 rows (status != PENDING), throws "Payment already EXPIRED". Webhook handler catches the exception, logs a warning, and returns 200 OK to SePay.

**Impact:** User paid money but order stays `PENDING_PAYMENT` forever. No fulfillment, no refund trigger.

**Fix:** Allow late webhook to mark payment as PAID if amount matches and within a grace period (e.g., 1 hour after expiry). Alternatively, create a reconciliation job that periodically checks EXPIRED payments against SePay transaction records.

---

### 3. Double-click "Confirm Order" creates duplicate orders

**Location:** `CartService.java:314` → `checkoutOrderId = "ORD-" + System.currentTimeMillis() + "-" + userId`

**Problem:** Each click generates a different `checkoutOrderId` (different millisecond timestamp). The idempotency check `findByUserIdAndCheckoutOrderId` doesn't help because each request has a unique ID. Both requests pass validation, both deduct stock, both create orders.

**Impact:** User sees 2 orders, stock deducted twice, potential double-charging.

**Fix:** Frontend: disable the confirm button immediately after first click. Backend: use a stable idempotency key for the entire checkout flow (e.g., hash of cart items + userId) instead of timestamp-based orderId.

---

### 4. WebhookLog saved before markAsPaid — lost payment on DB failure

**Location:** `WebhookService.java:104-117`

**Problem:** Current flow: check duplicate → save WebhookLog → call `markAsPaid`. If `markAsPaid` fails (e.g., DB connection lost after WebhookLog saved), the webhook is marked as "processed". On retry, SePay sends the same webhook → WebhookLog exists → returns success without retrying `markAsPaid`.

**Impact:** Payment never marked PAID. User paid, order stays `PENDING_PAYMENT`.

**Fix:** Save WebhookLog AFTER `markAsPaid` succeeds, or wrap both operations in a single database transaction.

---

### 5. RabbitMQ down → PaymentConfirmedEvent lost → order never PAID

**Location:** `WebhookService.java:154-158`

**Problem:** `rabbitTemplate.convertAndSend` is called directly without try-catch. If RabbitMQ is down, an `AmqpException` is thrown. The payment is already marked PAID in the DB, but the event is never published. SePay receives 500 and retries. On retry, WebhookLog already exists → skip processing → event never published.

**Impact:** Payment = PAID, but Order stays `PENDING_PAYMENT` (no event to trigger status transition). Cart is never cleared.

**Fix:** Wrap the `convertAndSend` call in a try-catch that logs the failure. Better: use `@TransactionalEventListener(phase = AFTER_COMMIT)` with an outbox pattern for guaranteed delivery.

---

### 6. No ownership check on createPayment

**Location:** `PaymentService.java:41-59`

**Problem:** `createPayment` does not verify that the requesting user owns the order. Any authenticated user with a valid JWT can create a payment for any orderId (if they can guess/enumerate the ID).

**Impact:** Security + accounting issue. User B could pay for User A's order, causing confusion in order ownership and financial records.

**Fix:** Add ownership verification in `createPayment`: call `orderClient.getOrderUserId(orderId)` and compare with the requesting user's ID.

## HIGH (8 issues)

### 7. Stock restoration silent failure

**Location:** `OrderService.java:190-200`

**Problem:** `restoreStockForOrder()` catches all exceptions and only logs. No retry, no compensation, no alerting.

**Impact:** If ProductService is down when an order is cancelled, stock is permanently lost. Order shows `CANCELLED` but inventory is incorrect.

**Fix:** Publish a `StockRestorationFailed` event to RabbitMQ with retry logic, or implement a compensation saga pattern.

---

### 8. Price drift between cart add and checkout

**Location:** `CartService.java:402-435` (`validateCartItems` checks stock but NOT price)

**Problem:** User adds item at price X. Price changes to Y in ProductService. Checkout proceeds with the old snapshot price X. OrderService calculates total from the snapshot. User pays the old (lower) price.

**Impact:** Revenue loss if prices increased. User confusion if prices decreased.

**Fix:** Add price validation in `validateCartItems()` or add a price re-check step in the checkout flow. Reject checkout if snapshot price differs from current price by more than a threshold.

---

### 9. Overpayment accepted / amount mismatch

**Location:** `WebhookService.java:250-254`

**Problem:** Webhook validation checks `transferAmount > 0` and `transferAmount >= payment.amount`. Overpayments are accepted. Underpayments with fuzzy matching could match the wrong payment.

**Impact:** Accounting discrepancies. Wrong payment could be marked PAID if amounts happen to match.

**Fix:** Require exact amount match. Reject both overpayments and underpayments.

---

### 10. Fuzzy paymentCode matching too loose

**Location:** `WebhookService.java:249`

**Problem:** `storedNormalized.contains(normalized) || normalized.contains(storedNormalized)` — substring match. Payment code `ABC123` could match webhook content `ABC123456` or vice versa.

**Impact:** Wrong payment could be matched and marked PAID.

**Fix:** Use exact match only. If fuzzy matching is needed for UX reasons, require a much stricter similarity threshold and additional validation (e.g., amount + timestamp proximity).

---

### 11. No circuit breakers on inter-service calls

**Location:** All Feign clients (`ProductClient`, `OrderClient`, etc.)

**Problem:** No circuit breaker configured. Feign clients use default timeouts (connect: 10s, read: 60s). If a downstream service is slow or down, threads block until timeout, eventually exhausting the thread pool.

**Impact:** Cascading failures. One slow service can bring down the entire system.

**Fix:** Add Resilience4j circuit breakers with fallback strategies to all inter-service Feign calls. Configure explicit timeouts (connect: 2s, read: 5s).

---

### 12. JWT expiry mid-checkout

**Location:** Frontend checkout flow (CheckoutPage.tsx)

**Problem:** No explicit handling. If JWT expires while user is filling the checkout form, the next API call returns 401. The partial checkout may have already deducted stock.

**Impact:** User loses their checkout progress. Stock may be partially deducted with no order created.

**Fix:** Implement JWT refresh token flow. Check token expiry before starting checkout and refresh if needed. Handle 401 gracefully with a retry after token refresh.

---

### 13. Unmatched webhook silently ignored

**Location:** `WebhookService.java:131-133`

**Problem:** If `resolvePaymentCode` returns null (no matching payment found), the webhook logs a warning and returns `true` (success to SePay). No record is kept for manual reconciliation.

**Impact:** Legitimate payment with garbled transfer content is silently ignored. No way to detect or recover.

**Fix:** Log unmatched webhooks to a separate `unmatched_payments` table for manual reconciliation. Alert on high volume of unmatched webhooks.

---

### 14. cancelPayment has no try-catch around RabbitMQ publish

**Location:** `PaymentService.java:126-130`

**Problem:** `cancelPayment` calls `rabbitTemplate.convertAndSend` directly without try-catch. If RabbitMQ is down, `AmqpException` is thrown → `@Transactional` rolls back → payment is NOT cancelled in DB.

**Impact:** User clicks "Cancel payment" → error returned → payment stays PENDING → stock stays deducted. Contrast with `OrderEventPublisher.publish()` which has proper try-catch.

**Fix:** Wrap the `convertAndSend` call in a try-catch that logs the failure but allows the DB transaction to commit:

```java
try {
    rabbitTemplate.convertAndSend(...);
} catch (AmqpException e) {
    log.warn("Failed to publish PaymentCancelledEvent: {}", e.getMessage());
}
```

## MEDIUM (1 issue)

### 15. PaymentPage has no alert when navigating away

**Location:** `PaymentPage.tsx:122-133` and `PaymentPage.tsx:178-185`

**Problem:** `beforeunload` only fires on tab close / page reload, NOT on React Router client-side navigation. The "Quay lại" button navigates to `/orders` without any warning or payment cancellation.

**Impact:** User navigates away thinking they cancelled, but payment stays PENDING. Stock remains deducted until the 5-minute expiry job runs.

**Fix:** Use `useBlocker` from React Router to show a confirmation dialog when navigating away from an active payment:

```tsx
const blocker = useBlocker(
  ({ currentLocation, nextLocation }) =>
    !paymentCompletedRef.current &&
    currentLocation.pathname !== nextLocation.pathname
);

useEffect(() => {
  if (blocker.state === "blocked") {
    const confirmed = window.confirm(
      "Bạn đang trong quá trình thanh toán. Nếu rời đi, đơn hàng sẽ bị hủy sau 5 phút. Bạn có chắc muốn rời đi?"
    );
    if (confirmed) blocker.proceed();
    else blocker.reset();
  }
}, [blocker.state]);
```

## Issues Marked as Incorrect (Retracted)

### ~Polling error stops after 3 retries~

**Original claim:** TanStack Query stops polling after 3 failures.

**Correction:** `retry: 3` applies to a single fetch attempt. The `refetchInterval: 3000` continues to trigger new fetches regardless of previous error state. Polling does NOT stop.

### ~No DLQ for payment event queues~

**Original claim:** Payment event queues have no dead letter queue configured.

**Correction:** DLQs are properly configured in `PaymentEventQueueConfig.java`:
- `payment.confirmed.queue` → DLX → `payment.confirmed.dlq`
- `payment.expired.queue` → DLX → `payment.expired.dlq`
- `payment.cancelled.queue` → DLX → `payment.cancelled.dlq`

Note: `x-message-ttl: 300000` (5 min) is on the main queue, not the DLQ. Messages that fail processing for > 5 minutes move to DLQ where they sit indefinitely (no DLQ consumer).
