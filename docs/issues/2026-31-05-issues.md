**Scope:** Stock deduction compensation gap in OrderService checkout flow

## 2. The Remaining Gap: No Compensation on Order Creation Failure

### 2.1 The Problem

```
OrderService.createOrderFromCheckout() [@Transactional]
  → Step 1: deductStockForOrder()
      → ProductClient.batchDeductStock() [Feign + Resilience4j]
      → ProductService → DynamoDB TransactWriteItems ✅ (stock deducted atomically)
  → Step 2: orderRepository.save(order)
      → ❌ DataIntegrityViolationException (duplicate key, invalid FK, DB full)
  → Step 3: @Transactional rolls back
      → Local DB transaction undone ✅
      → BUT DynamoDB stock deduction is NOT rolled back ❌
  → Result: Stock permanently lost, no order created
```

**Why DynamoDB atomicity doesn't help:** `TransactWriteItems` guarantees all-or-nothing **within DynamoDB**. It does NOT coordinate with MariaDB. If the DynamoDB transaction succeeds and MariaDB fails, the two databases are inconsistent.

### 2.2 Code Analysis

**File:** `OrderService/src/main/java/com/iuh/fit/service/OrderService.java`

```java
@Transactional                                          // Line 55
public OrderResponse createOrderFromCheckout(...) {
    // Idempotency check
    Order existing = orderRepository
            .findByUserIdAndCheckoutOrderId(userId, request.getOrderId())
            .orElse(null);
    if (existing != null) {
        return toResponse(existing);
    }

    // Total validation
    BigDecimal calculatedTotal = calculateTotal(request.getItems());
    // ...

    // STEP 1: Stock deduction (OUTSIDE local transaction scope, but INSIDE @Transactional)
    deductStockForOrder(request.getItems());             // Line 72

    // STEP 2: Build and save order
    Order order = new Order();
    order.setStatus(OrderStatus.PENDING_PAYMENT);
    // ...

    try {
        Order saved = orderRepository.save(order);       // Line 109 — CAN THROW
        publishOrderCreatedEvent(saved);
        return toResponse(saved);
    } catch (DataIntegrityViolationException ex) {       // Line 113
        // Only handles duplicate — does NOT compensate stock
        Order duplicated = orderRepository
                .findByUserIdAndCheckoutOrderId(userId, request.getOrderId())
                .orElseThrow(() -> ex);
        return toResponse(duplicated);
    }
    // If any OTHER exception is thrown (not DataIntegrityViolationException),
    // @Transactional rolls back BUT stock is already deducted.
}
```

**The catch block at line 113 only handles the idempotent duplicate case.** It does not call `batchRestoreStock()`. Any other exception (FK violation, DB connection lost, constraint violation) causes the transaction to roll back without compensating the stock deduction.

### 2.3 How Likely

| Scenario | Likelihood | Trigger |
|---|---|---|
| Database constraint violation | Low | Duplicate order ID race condition, invalid foreign key |
| MariaDB connection lost mid-transaction | Low | Network blip, DB restart, connection pool exhaustion |
| MariaDB disk full | Low-Medium | Unmonitored disk, large data growth |
| Feign timeout after successful DynamoDB write | Low | Network timeout between ProductService response and OrderService receipt |

**Cumulative risk:** Low per-occurrence, but **high impact** when it happens (permanent inventory loss, no detection).

### 2.4 Business Consequence

- Product stock shows fewer units than reality
- Over time, popular items appear "sold out" when they're not
- No automated detection — inconsistency is silent
- Customer support gets no alerts, admin has no visibility

## 3. Secondary Gap: Stock Restoration is Fire-and-Forget on Cancellation

### 3.1 The Problem

When an order is cancelled, `batchRestoreStock()` is called but **all errors are silently swallowed**:

**File:** `OrderService/src/main/java/com/iuh/fit/service/OrderService.java`

```java
@CircuitBreaker(name = "productService", fallbackMethod = "restoreStockFallback")
@Retry(name = "productService")
@Bulkhead(name = "productService")
public void batchRestoreStock(List<BatchStockRequest> items) {
    try {
        BatchStockResponse response = productClient.batchRestoreStock(items);
        if (!response.success() && response.failedItems() != null) {
            String failedDetails = response.failedItems().stream()
                    .map(f -> f.variantId() + ": " + f.reason())
                    .collect(Collectors.joining(", "));
            log.error("Stock restoration partially failed: {}", failedDetails);
        }
    } catch (Exception e) {
        log.error("Stock restoration failed: {}", e.getMessage());
    }
    // ALL exceptions swallowed — no rethrow, no alerting, no retry queue
}

public void restoreStockFallback(List<BatchStockRequest> items, Throwable t) {
    log.error("Stock restoration failed after retries: {}", t.getMessage());
    // ALSO swallowed — fire-and-forget logging only
}
```

### 3.2 Business Consequence

- If ProductService is down during order cancellation, stock is never restored
- No retry mechanism, no DLQ, no alerting
- Inventory permanently reserved for cancelled orders

## 4. Tertiary Gap: No Idempotency Key on Stock Deduction

### 4.1 The Problem

`ProductClient.batchDeductStock()` has no idempotency key. Resilience4j is configured with:

```properties
resilience4j.retry.instances.productService.maxAttempts=3
resilience4j.retry.instances.productService.waitDuration=1s
resilience4j.retry.instances.productService.retryExceptions=feign.FeignException$ServiceUnavailable,feign.FeignException$GatewayTimeout
```

If a `ServiceUnavailable` or `GatewayTimeout` occurs **after** ProductService successfully processed the request but **before** the response reached OrderService, Resilience4j retries. ProductService processes the same request again → **double deduction**.

### 4.2 How Likely

Low but real. Happens during:
- Network timeout between services
- ProductService restart mid-request
- Load balancer health check flapping

## 6. Recommended Fixes

### Fix 1: Compensate Stock on Order Creation Failure (Priority: HIGH)

Wrap the order save in a try-catch that calls `batchRestoreStock()` on failure:

```java
@Transactional
public OrderResponse createOrderFromCheckout(...) {
    // ... idempotency check, validation ...

    deductStockForOrder(request.getItems());

    try {
        Order order = buildOrder(...);
        Order saved = orderRepository.save(order);
        publishOrderCreatedEvent(saved);
        return toResponse(saved);
    } catch (DataIntegrityViolationException ex) {
        // Idempotent duplicate — safe to return existing
        Order duplicated = orderRepository
                .findByUserIdAndCheckoutOrderId(userId, request.getOrderId())
                .orElseThrow(() -> ex);
        return toResponse(duplicated);
    } catch (Exception ex) {
        // Any other failure — COMPENSATE stock before rethrowing
        log.error("Order creation failed, restoring stock: {}", ex.getMessage());
        try {
            restoreStockForOrder(request.getItems());
        } catch (Exception restoreEx) {
            log.error("Stock restoration ALSO failed — manual intervention required", restoreEx);
            // Consider: save to outbox/DB for later reconciliation
        }
        throw ex; // Let @Transactional roll back
    }
}
```

### Fix 2: Publish Stock Restoration Failure Event (Priority: MEDIUM)

Instead of swallowing errors in `batchRestoreStock()`, publish a `STOCK_RESTORE_FAILED` event:

```java
public void batchRestoreStock(List<BatchStockRequest> items) {
    try {
        BatchStockResponse response = productClient.batchRestoreStock(items);
        if (!response.success()) {
            // Publish event for async retry
            eventPublisher.publishStockRestoreFailed(items, response.failedItems());
        }
    } catch (Exception e) {
        eventPublisher.publishStockRestoreFailed(items, e);
    }
}
```

### Fix 3: Add Idempotency Key to Stock Deduction (Priority: MEDIUM)

Pass an idempotency key (e.g., `orderId` or `checkoutOrderId`) with the stock deduction request. ProductService deduplicates based on this key:

```java
// OrderService
BatchStockRequest request = new BatchStockRequest(items, request.getOrderId()); // idempotency key

// ProductService
async function batchDeductStock(items, idempotencyKey) {
    // Check if already processed
    const existing = await idempotencyStore.get(idempotencyKey);
    if (existing) return existing;

    // Process and store result
    const result = await dynamoClient.send(new TransactWriteCommand({...}));
    await idempotencyStore.set(idempotencyKey, result, { ttl: 3600 });
    return result;
}
```

## 7. Outbox Gap: PaymentExpiryJob Bypasses Outbox Pattern

### 7.1 The Problem

`PaymentExpiryJob` publishes `PAYMENT_EXPIRED` events **directly** via `rabbitTemplate.convertAndSend()` at lines 78-82, completely bypassing the outbox table. This is the **only event type** in the entire system that does so — all other events (`ORDER_CREATED`, `ORDER_PAID`, `PAYMENT_CONFIRMED`, `PAYMENT_CANCELLED`) correctly route through the outbox when `feature.outbox.enabled=true`.

**File:** `PaymentService/src/main/java/iuh/fit/PaymentService/service/PaymentExpiryJob.java`

```java
// Lines 70-83 — DIRECT publish, NO outbox check, NO error handling
PaymentExpiredEvent event = new PaymentExpiredEvent(
        refreshed.getId(),
        refreshed.getOrderId(),
        refreshed.getCheckoutOrderId(),
        refreshed.getPaymentCode(),
        refreshed.getAmount(),
        Instant.now()
);
rabbitTemplate.convertAndSend(
        RabbitMQConfig.PAYMENT_EXCHANGE,
        RabbitMQConfig.PAYMENT_EXPIRED_ROUTING_KEY,
        event
);
log.info("Published PaymentExpiredEvent for payment: {}", refreshed.getId());
```

**Compare with `PaymentService.markAsPaid()`** (correct pattern):

```java
// PaymentService.java:190-211 — checks outboxEnabled, saves to DB
private void publishEvent(Object event, String eventType, Long aggregateId,
                          String exchange, String routingKey) {
    if (outboxEnabled) {
        String payload = objectMapper.writeValueAsString(event);
        OutboxEvent outboxEvent = OutboxEvent.builder()
                .eventType(eventType)
                .eventId(UUID.randomUUID().toString())
                .aggregateId(aggregateId)
                .payload(payload)
                .exchange(exchange)
                .routingKey(routingKey)
                .build();
        outboxRepository.save(outboxEvent);  // Saved in same transaction
    } else {
        rabbitTemplate.convertAndSend(exchange, routingKey, event);
    }
}
```

### 7.2 Full Execution Flow

```
PaymentExpiryJob.expirePendingPayments()
  [@Scheduled(fixedRate = 60000), @SchedulerLock, @Transactional]

  Step 1: Query PENDING payments that have expired
    → paymentRepository.findByStatus(PENDING) → filter by expiresAt < now()
    → Up to 1000 payments per run

  Step 2: Bulk mark as EXPIRED
    → paymentRepository.expirePendingPayments(Instant.now())
    → UPDATE payment SET status = 'EXPIRED' WHERE status = 'PENDING' AND expires_at <= ?
    → ✅ Committed within @Transactional

  Step 3: Re-fetch & race-condition check
    → For each payment, findById() to get fresh status
    → Skip if PAID (webhook won the race)
    → Skip if not EXPIRED (another process changed it)

  Step 4: Publish PAYMENT_EXPIRED event
    → rabbitTemplate.convertAndSend() DIRECT — NO OUTBOX ❌
    → If RabbitMQ down → event LOST
    → No try-catch around this call → exception propagates
    → @Transactional rolls back → payment status reverts to PENDING
    → Next run (60s later) tries again → infinite loop if RabbitMQ stays down
```

### 7.3 Failure Scenarios

| Scenario | What Happens | Consequence |
|---|---|---|
| **RabbitMQ down during publish** | `AmqpConnectException` thrown → `@Transactional` rolls back → payment stays PENDING | Order never cancelled, stock never restored. Job retries every 60s indefinitely. |
| **RabbitMQ restart mid-publish** | Connection lost → same as above | Same as above. |
| **Network partition (brief)** | Timeout → exception → rollback → retry next run | Self-healing if partition resolves within 60s. |
| **RabbitMQ accepts but OrderService consumer fails** | Event delivered → consumer throws → goes to DLQ → **no DLQ consumer** | Event in `payment.expired.dlq` forever. Order stuck PENDING_PAYMENT. |

### 7.4 Business Consequence

| Entity | State After Failure | Impact |
|---|---|---|
| Payment | `PENDING` (rolled back) or `EXPIRED` (if exception after commit) | Inconsistent — depends on exact failure point |
| Order | `PENDING_PAYMENT` (never receives PAYMENT_EXPIRED) | User sees "waiting for payment" indefinitely |
| Product stock | Deducted (never restored) | Inventory permanently reserved |
| User experience | Confusing — order appears active but payment expired | Support tickets, trust erosion |

### 7.5 How Likely

| Trigger | Frequency | Notes |
|---|---|---|
| RabbitMQ restart during deployment | 1-2x/week | Brief window (5-15s) |
| RabbitMQ crash (OOM, disk full) | Rare but possible | Recovery takes 30s-2min |
| Network partition in Docker | Low | More likely in production than dev |
| Connection pool exhausted | Under high load | All RabbitMQ operations fail |

**Cumulative risk:** Medium. The job has ShedLock (`lockAtMostFor = "55s"`) which prevents duplicate runs, but does NOT protect against message loss.

### 7.6 Recommended Fix

Route `PAYMENT_EXPIRED` through outbox, same as all other events:

```java
// PaymentExpiryJob.java — replace lines 78-83 with:

// Option A: Inject OutboxEventRepository and save directly
OutboxEvent outboxEvent = OutboxEvent.builder()
        .eventType("PAYMENT_EXPIRED")
        .eventId(UUID.randomUUID().toString())
        .aggregateId(refreshed.getId())
        .payload(objectMapper.writeValueAsString(event))
        .exchange(RabbitMQConfig.PAYMENT_EXCHANGE)
        .routingKey(RabbitMQConfig.PAYMENT_EXPIRED_ROUTING_KEY)
        .build();
outboxRepository.save(outboxEvent);

// OutboxPublisherJob will pick it up and publish when RabbitMQ is available.
// Since this is inside @Transactional, if the transaction rolls back,
// the outbox row also rolls back — correct behavior (payment stays PENDING, retry next run).
```

**Why this works:**
- If RabbitMQ is down: outbox row saved → `OutboxPublisherJob` retries every 5s until RabbitMQ recovers
- If transaction rolls back: outbox row also rolls back → payment stays PENDING → next run retries
- No event loss, no infinite loop, no manual intervention needed

## 8. Outbox Gap: No Retry Backoff on OutboxPublisherJob

### 8.1 The Problem

`OutboxPublisherJob` retries failed events with a **constant 5-second interval** and **no exponential backoff**, **no jitter**, and **no differentiation** between transient and permanent failures.

**File:** `OrderService/src/main/java/com/iuh/fit/service/OutboxPublisherJob.java`

```java
// Line 35 — hardcoded 5s, not configurable
@Scheduled(fixedDelay = 5000)
public void publishPendingEvents() {
    // ...
    for (OutboxEvent event : events) {
        publishSingleEvent(event);
    }
}

// Lines 54-74 — retry logic
@Transactional
public void publishSingleEvent(OutboxEvent event) {
    try {
        rabbitTemplate.convertAndSend(event.getExchange(), event.getRoutingKey(), event.getPayload());
        event.setStatus("PUBLISHED");
        event.setPublishedAt(Instant.now());
        outboxRepository.save(event);
    } catch (Exception e) {
        event.setRetryCount(event.getRetryCount() + 1);
        event.setErrorMessage(e.getMessage());
        if (event.getRetryCount() >= event.getMaxRetries()) {
            event.setStatus("FAILED");
            outboxRepository.save(event);
        } else {
            outboxRepository.save(event);
            // No delay calculation, no backoff — just save and wait for next tick
        }
    }
}
```

**Identical code in PaymentService:** `PaymentService/src/main/java/iuh/fit/PaymentService/service/OutboxPublisherJob.java`

### 8.2 Retry Timeline (Current Behavior)

```
t=0s:    Event publish FAIL (RabbitMQ restarting)  → retryCount=1
t=5s:    Event publish FAIL (RabbitMQ still down)   → retryCount=2
t=10s:   Event publish FAIL (RabbitMQ still down)   → retryCount=3
t=15s:   Event publish FAIL (RabbitMQ still down)   → retryCount=4
t=20s:   Event publish FAIL (RabbitMQ still down)   → retryCount=5 → status=FAILED ❌

Total: 20 seconds, 5 retries burned, event marked FAILED permanently.
RabbitMQ recovers at t=25s — but event is already FAILED, no more retries.
```

### 8.3 Two Failure Types, One Retry Strategy

| Failure Type | Examples | Should Retry? | Ideal Strategy |
|---|---|---|---|
| **Transient** | RabbitMQ restart, network blip, connection timeout, TLS handshake failure | ✅ Yes | Exponential backoff: 5s → 10s → 20s → 40s → 80s. Wait for service recovery. |
| **Permanent** | Wrong routing key, deleted exchange, invalid payload, serialization error | ❌ No (or very few) | Fail fast, alert immediately. Don't burn retries. |

**Current behavior treats both identically:** constant 5s retry, burns all 5 retries in 20s regardless of failure type.

### 8.4 Consequences

| Consequence | Detail |
|---|---|
| **Premature failure** | Transient failures (RabbitMQ restart takes 10-30s) burn all retries before service recovers |
| **Thundering herd** | All failed events retry at the same 5s tick → spike in RabbitMQ load when it comes back |
| **No jitter** | No randomization → synchronized retries across all events |
| **Log spam** | Permanent failures generate error logs every 5s for 20s (5 log lines per event) |
| **Wasted resources** | Each retry acquires DB connection, serializes payload, attempts RabbitMQ connection |

### 8.5 How Likely

| Scenario | Impact with Current Retry |
|---|---|
| RabbitMQ restart (10-15s) | Burns 2-3 retries, may survive if 2 retries remain |
| RabbitMQ crash + manual recovery (1-5 min) | **Always FAILED** — 5 retries exhausted in 20s, no retries left when RabbitMQ recovers |
| Wrong routing key (deploy bug) | Burns 5 retries in 20s, then FAILED. At least stops eventually. |
| Brief network blip (1-2s) | Survives — retry 2 succeeds |

### 8.6 Recommended Fix

Add exponential backoff with jitter. Two approaches:

**Option A: Add `retry_after` column (cleanest)**

```sql
ALTER TABLE outbox_events ADD COLUMN retry_after TIMESTAMP NULL;
```

```java
// OutboxEventRepository.java
@Query("SELECT e FROM OutboxEvent e WHERE e.status = 'PENDING' " +
       "AND (e.retryAfter IS NULL OR e.retryAfter <= CURRENT_TIMESTAMP) " +
       "ORDER BY e.createdAt ASC")
List<OutboxEvent> findRetryableEvents(Pageable pageable);
```

```java
// OutboxPublisherJob.java
@Transactional
public void publishSingleEvent(OutboxEvent event) {
    try {
        rabbitTemplate.convertAndSend(event.getExchange(), event.getRoutingKey(), event.getPayload());
        event.setStatus("PUBLISHED");
        event.setPublishedAt(Instant.now());
        event.setRetryAfter(null);
        outboxRepository.save(event);
    } catch (Exception e) {
        event.setRetryCount(event.getRetryCount() + 1);
        event.setErrorMessage(e.getClass().getSimpleName() + ": " + e.getMessage());

        if (event.getRetryCount() >= event.getMaxRetries()) {
            event.setStatus("FAILED");
        } else {
            // Exponential backoff: 5s * 2^retryCount + jitter(0-2s)
            long baseDelayMs = 5000;
            long exponentialDelay = baseDelayMs * (long) Math.pow(2, event.getRetryCount() - 1);
            long jitter = ThreadLocalRandom.current().nextLong(0, 2000);
            long totalDelayMs = Math.min(exponentialDelay + jitter, 300_000); // Cap at 5 min
            event.setRetryAfter(Instant.now().plusMillis(totalDelayMs));
        }
        outboxRepository.save(event);
    }
}
```

**Retry timeline with backoff:**

```
t=0s:    FAIL → retryCount=1, retry_after=t+5s
t=5s:    FAIL → retryCount=2, retry_after=t+10s
t=15s:   FAIL → retryCount=3, retry_after=t+20s
t=35s:   FAIL → retryCount=4, retry_after=t+40s
t=75s:   FAIL → retryCount=5, retry_after=t+80s
t=155s:  FAIL → status=FAILED (2min 35s total, vs 20s before)
```

This gives RabbitMQ ~2.5 minutes to recover instead of 20 seconds.

**Option B: Simpler — classify errors**

```java
private boolean isTransientError(Exception e) {
    return e instanceof AmqpConnectException
        || e instanceof AmqpTimeoutException
        || e instanceof ResourceAccessException;
}

@Transactional
public void publishSingleEvent(OutboxEvent event) {
    try {
        rabbitTemplate.convertAndSend(...);
        // ... success
    } catch (Exception e) {
        if (!isTransientError(e)) {
            // Permanent error — fail immediately, don't waste retries
            event.setStatus("FAILED");
            event.setErrorMessage("PERMANENT: " + e.getMessage());
            outboxRepository.save(event);
            return;
        }
        // Transient error — retry with backoff
        // ... same as Option A
    }
}
```

## 9. Outbox Gap: No Cleanup of Old Published/Failed Events

### 9.1 The Problem

There is **no job, no endpoint, no policy** to delete or archive events with status `PUBLISHED` or `FAILED`. The `outbox_events` table grows unbounded.

**Entity has `createdAt` but nobody uses it for cleanup:**

```java
// OutboxEvent.java
@CreationTimestamp
@Column(name = "created_at", updatable = false)

```

**Repository has no delete methods:**


### 9.6 Recommended Fix

**Option A: Scheduled cleanup with retention policy (recommended)**

```java
@Service
public class OutboxCleanupJob {

    private static final Logger log = LoggerFactory.getLogger(OutboxCleanupJob.class);
    private static final int RETENTION_DAYS = 30;

    private final OutboxEventRepository outboxRepository;

    public OutboxCleanupJob(OutboxEventRepository outboxRepository) {
        this.outboxRepository = outboxRepository;
    }

    @Scheduled(cron = "0 0 2 * * *")  // 2:00 AM daily
    @Transactional
    public void cleanupOldPublishedEvents() {
        Instant cutoff = Instant.now().minus(Duration.ofDays(RETENTION_DAYS));
        int deleted = outboxRepository.deleteByStatusAndCreatedAtBefore("PUBLISHED", cutoff);
        if (deleted > 0) {
            log.info("Cleaned up {} published outbox events older than {} days", deleted, RETENTION_DAYS);
        }
    }

    @Scheduled(cron = "0 30 2 * * *")  // 2:30 AM daily (after published cleanup)
    @Transactional
    public void archiveOldFailedEvents() {
        Instant cutoff = Instant.now().minus(Duration.ofDays(RETENTION_DAYS));
        int archived = outboxRepository.archiveByStatusAndCreatedAtBefore("FAILED", cutoff);
        if (archived > 0) {
            log.warn("Archived {} failed outbox events older than {} days — review needed", archived, RETENTION_DAYS);
        }
    }
}
```

**Why this approach:**
- `PUBLISHED` events: safe to DELETE after 30 days (already delivered, no action needed)
- `FAILED` events: ARCHIVE instead of DELETE (need investigation, don't lose data)
- Runs at 2 AM (low traffic window)
- Batch delete via single query (efficient)

**Option B: Add `@Scheduled` with batch delete (safer for large tables)**

```java
@Scheduled(cron = "0 0 2 * * *")
@Transactional
public void cleanupOldEvents() {
    Instant cutoff = Instant.now().minus(Duration.ofDays(RETENTION_DAYS));
    int totalDeleted = 0;
    int batchSize = 1000;

    while (true) {
        int deleted = outboxRepository.deleteBatch("PUBLISHED", cutoff, PageRequest.of(0, batchSize));
        totalDeleted += deleted;
        if (deleted < batchSize) break;  // No more rows to delete
    }

    if (totalDeleted > 0) {
        log.info("Cleaned up {} published outbox events in batches", totalDeleted);
    }
}
```

This avoids long-running transactions and lock contention on large tables.
